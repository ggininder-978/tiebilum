# Goal-Driven Verification Spec: 目標驅動驗證規格

將「指令式」命令轉化為「可驗證」目標的標準操作流程。

## 1. 任務轉化模型
| 弱指令 (指令式) | 強目標 (聲明式/可驗證) |
| :--- | :--- |
| 「增加欄位驗證」 | 「為無效輸入撰寫測試，並讓其通過。」 |
| 「修復 Bug」 | 「撰寫重現 Bug 的測試，實作修復後讓其通過。」 |
| 「優化效能」 | 「定義目前的基準(Baseline)，設定目標值並驗證達成。」 |

## 2. 驗證循環 (The Loop)
對於多步驟任務，必須明確列出階段性驗證點：
1.  **步驟 A** -> 驗證方式 (e.g., `ls` 檢查文件, `curl` 測試接口)。
2.  **步驟 B** -> 驗證方式 (e.g., 執行單元測試)。
3.  **步驟 C** -> 驗證方式 (e.g., 檢查 Git Log 提交狀態)。

## 3. 測試先行 (Test-First)
*   在修復 Bug 前，必須先寫一個**會失敗的測試**來證明 Bug 存在。
*   如果修復後測試通過，且舊有測試未失敗，方可宣告成功。
